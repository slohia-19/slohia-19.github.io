Margin

A VS Code extension that uses AI to make code readable for designers and PMs — without requiring an engineering background.

Hackathon Sprint Role: Product Manager VS Code Extension API · OpenAI API · Cursor · Claude
The Problem

Designers and PMs who work alongside engineers struggle to parse code — understanding what a component does, how it's structured, or why it was written a certain way.

My Role

PM on a team of designers and PMs — problem definition, MVP scoping, feature prioritization, and roadmap planning under a tight hackathon time constraint.

The Team

Designers and product managers — none of us had built a VS Code extension before. First hackathon for the whole team.

Outcome

Shipped a functional VS Code extension with AI-powered code annotations within a single hackathon sprint.

Background & Context

Designers increasingly work in or alongside codebases — reviewing pull requests, inspecting component files, or trying to understand why something looks the way it does. But without an engineering background, even well-written code can feel opaque.

At our first hackathon, our team of designers and product managers set out to build something for ourselves: a tool that lets designers engage meaningfully with code without needing to fully learn it first.

The Problem

Designers and PMs who work closely with engineering teams often find themselves staring at code they can't fully parse — understanding what a component does, how it's structured, or why it was written a certain way. Margin was built to close that gap.

Problem Statement: Designers and PMs need a way to understand code in context because the gap between how engineers write code and how non-engineers can read it creates friction, slows collaboration, and leaves designers dependent on engineers to interpret their own files.

What Margin Does

Margin lives inside VS Code as an extension. Users toggle it on and it immediately goes to work — visually annotating the code in front of them. The result is a layer of comprehension on top of existing code, without modifying the code itself.

Code chunk highlighting — Margin groups and highlights sections of code by functionality, giving users a visual sense of how the file is structured at a glance.

AI-powered hover descriptions — Hovering over a highlighted chunk surfaces a plain-language explanation of what that code does, contextualized within the broader project.

How We Built It

We built Margin as a native VS Code extension and integrated the OpenAI API to handle code parsing and generate natural language explanations. The extension reads the active file, segments it into logical chunks, and feeds those chunks to the model to produce context-aware descriptions on demand.

Stack
  • VS Code Extension API
  • OpenAI API
  • Cursor
  • Claude
How It Works
  • Reads the active file in VS Code
  • Segments it into logical code chunks
  • Feeds chunks to the model for context-aware descriptions
  • Surfaces explanations on hover — no code modification

Challenges

Building Margin was harder than any of us anticipated — in the best possible way. None of us had built a VS Code extension before, and integrating with the OpenAI API was a steep learning curve. We spent meaningful time just figuring out how the API worked before we could apply it toward what we actually wanted to build.

There was a certain irony in wishing Margin existed while we were struggling to build it — we could have really used an AI explanation of our own extension code mid-build.

What We're Proud Of

This was our first hackathon. Our primary goal going in was simple: build something real and usable. We did that.

Beyond the output, we're proud of how we navigated ambiguity as a team — defining the problem, scoping the MVP, and shipping something functional within a tight time constraint. We also came away with a genuine appreciation for AI as a development accelerator; tools like Cursor and Claude made it possible for a team without deep engineering backgrounds to build something this technical.

What's Next

Margin is a foundation, not a finished product. The roadmap focuses on deepening the comprehension layer and improving the overall experience.

01

Information architecture view — a high-level visual map of how a codebase is organized, helping designers understand structure before diving into individual files.

02

Code heatmaps — visual indicators of complexity, frequency of change, or other signals that help designers know where to focus.

03

Design & UX improvements — polish the extension interface to feel as considered as the products our users design themselves.

Learnings & Takeaways

  • New technical skills are learnable under pressure. We each learned how to build a VS Code extension from scratch — a genuinely new skill for all of us, acquired in a single sprint.
  • Scope management is a product skill. With limited time, we had to make fast, deliberate tradeoffs — deciding which features made the MVP and which could wait. That discipline shaped the product significantly.
  • AI is a real development accelerator. Tools like Cursor and Claude made it possible for a team without deep engineering backgrounds to ship something genuinely technical. We experienced firsthand what AI-assisted development enables.
  • Build for yourself first. The clearest signal that we were solving a real problem was that we wanted the tool ourselves — and felt its absence while building it.
